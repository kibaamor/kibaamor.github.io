数据库和缓存备忘

一、 redis 和 memcached

1.1 redis 的一些技巧

  1. 易读性的前提下，key 设置尽可能小，减少资源的占用。
    redis value 可以用 int 就不要用string，对于小于 N 的 value，redis 内部有 shared_object 缓存。
  2. 拆分 key。
    主要是用在 redis 使用 hashes 情况下。同一个 hashes key 会落到同一个 redis 节点，hashes 过大的情况下会导致内存及请求分布的不均匀。考虑对 hash 进行拆分为小的 hash，使得节点内存均匀及避免单节点请求热点。
  3. 空缓存设置。
    对于部分数据，可能数据库始终为空，这时应该设置空缓存，避免每次请求都缓存 miss 直接打到 DB。
  4. 读失败后（比如 network timeout， 不是 cache miss）的写缓存策略（降级后一般读失败不触发回写缓存）。
  5. 序列化使用 protobuf，尽可能减少 size。
  6. 增量更新一致性：先用 EXPIRE、在用 ZADD/HSET 等，保证索引结构体务必存在的情况下去操作新增数据。
  7. BITSET: 存储每日登陆用户，单个标记位置（boolean），为了避免单个 BITSET 过大或者热点，需要使用 region sharding，比如按照 mid 求余 % 和 /10000，商为 KEY、余数作为offset。
  8. List：抽奖的奖池、顶弹幕，用于类似 Stack PUSH/POP 操作。
  9. Sortedset: double 类型。翻页、排序、有序的集合，杜绝 zrange 或者 zrevrange 返回的集合过大。
  10. Hashs: 过小的时候会使用压缩列表、过大的情况容易导致 rehash 内存浪费，也杜绝返回hgetall，对于小结构体，建议直接使用 memcache KV。
  11. String: SET 的 EX/NX 等 KV 扩展指令，SETNX 可以用于分布式锁、SETEX 聚合了SET + EXPIRE。
  12. Sets: 类似 Hashs，无 Value，去重等。
  13. 尽可能的 PIPELINE 指令，但是避免集合过大。
  14. 避免超大 Value。

1.2 memcached 的一些技巧

  1. flag 使用：标识 compress、encoding、large value 等；
  2. memcache 支持 gets，尽量读取，尽可能的 pipeline，减少网络往返；
  3. 使用二进制协议，支持 pipeline delete，UDP 读取、TCP 更新；

1.3 redis 对比 memcached

redis 和 memcached 的最大区别是 redis 单线程， memcached 多线程，所以 QPS 可能两者差异不大，但是吞吐会有很大的差别，比如大数据 value 返回的时候，redis 的 QPS 会抖动下降的的很厉害，因为单线程工作，其他查询进不来。所有建议纯 kv 走 memcached 。

redis 6 已经支持了多线程的网络 IO 。

memcached 使用 slab 方式做内存管理，存在一定的浪费。如果有大量接近的 item，建议调整 memcache 参数来优化每一个 slab 增长的 ratio ，可以通过设置 slab_automove 和 slab_reassign 开启 memcached 的动态/手动 move slab，防止某些 slab 热点导致内存足够的情况下引发 LRU 。

对于 redis 中频繁的 HGETALL 等批量操作（比如：获取一个人所有关注的人的 ID ），可以在 redis 前加一层 memcached 来缓存结果，形成 memcached + redis 双缓存设计。

三、缓存数据分片

  1. key 进行 hash 后取余

  2. 一致性 hash

    添加虚拟节点解决服务节点少时数据倾斜的问题。

  3. 有界负载一致性 hash

redis cluster 的做法：

1.把 16384 槽按照节点数量进行平均分配，由节点进行管理
2.对每个 key 按照 CRC16 规则进行 hash 运算
3.把 hash 结果对 16383 进行取余
4.把余数发送给 redis 节点
5.节点接收到数据，验证是否在自己管理的槽编号的范围
如果在自己管理的槽编号范围内，则把数据保存到数据槽中，然后返回执行结果
如果在自己管理的槽编号范围外，则会把数据发送给正确的节点，由正确的节点来把数据保存在对应的槽中

16384 = (1<<14) 。

Redis Cluster的节点之间会共享消息，每个节点都会知道是哪个节点负责哪个范围内的数据槽

高可用Redis(十二)：Redis Cluster

四、数据一致性

以 MySQL 和 Redis 为例，实现缓存和数据库的最终一致性。

  1. 读操作流程

    1. 从 Redis 读取数据，如果有就直接返回；
    2. 如果 Redis 中没有数据，从 MySQL 中读取数据；
    3. 将结果使用 SETNX 回填到 Redis 中，并返回数据。

  2. 写操作流程

    1. 直接更新 MySQL ；

    2. 删除或更新 Redis ；

      用删除操作可能会导致缓存穿透问题，对热点数据不建议使用删除操作。

      用更新操作可能会有 ABA 的问题，如：两次更新 v1 和 v2 ，缓存中的值会出现 v1 -> v2 -> v1 -> v2 。

    3. 使用 Canal 解析 MySQL 的 binlog，将解析的结果投递到一个消息队列中（如：kafka），然后再消费该消息队列更新 Redis 来保证最终一致性。

五、多级缓存

当采用多级缓存时，最重要是保证多级缓存的一致性：

  1. 清理的优先级是有要求的，先优先清理下游再上游；
  2. 下游的缓存 expire 要大于上游，里面穿透回源；

六、热点缓存

对于热点缓存 Key，按照如下思路解决：

  1. 小表广播，从 RemoteCache 提升为 LocalCache ， App 定时更新，甚至可以让运营平台支持广播刷新 LocalCache ；
  2. 主动监控防御预热，比如直播房间页高在线情况下直接外挂服务主动防御；
  3. 基础库框架支持热点发现，自动短时的 short-live cache ；
  4. 多 Cluster 支持；
  5. 多 Key 设计: 使用多副本，减小节点热点的问题；
    使用多副本 ms_1 ， ms_2 ， ms_3 每个节点保存一份数据，使得请求分散到多个节点，避免单点热点问题。

  6. 对于非常热的数据而且能够接收稍微过期一点的数据，可以不真正的删除数据，而是将设置该数据设置一个比较短的过期时间（比如 10 秒）。

    最好通过改 Redis 源码来实现，因为要高效的处理边缘情况（比如：该数据本身的过期时间已经比 10 秒小了)。

七、缓存穿透问题

  1. singlefly

    对关键字进行一致性 hash ，使其某一个维度的 key 一定命中某个节点，然后在节点内使用互斥锁，保证归并回源，但是对于批量查询无解。

  2. 分布式锁

    设置一个 lock key ，有且只有一个人成功，并且返回，交由这个人来执行回源操作，其他候选者轮训 cache 这个 lock key ，如果不存在去读数据缓存，hit 就返回， miss 继续抢锁。

  3. 队列

    如果 cache miss ，交由队列聚合一个 key ，来 load 数据回写缓存，对于 miss 当前请求可以使用 singlefly 保证回源。适合回源加载数据重的任务，比如评论 miss 只返回第一页，但是需要构建完成评论数据索引。

  4. lease

    第一个 cache miss 的人可以获得一个 lease ，这个 lease 与请求的 key 绑定。每个 key 隔固定时间分配一次（如 10 秒） lease 。持有 lease 的人可以去访问数据库，没有持有的人则需睡眠等待然后重新访问缓存等。这样就可以解决缓存穿透的问题。

    同时，在从数据库获得数据后，更新 cache 时验证 lease 则可以一定程度上保存数据一致性。
